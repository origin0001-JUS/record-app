> **핵심 변경**: 사용자가 옵션을 고르는 게 아니라, AI가 자료를 분석해서 맞는 옵션을 추천한다.
>
> **CEO Review 완료** (2026-04-10) — SELECTIVE EXPANSION, Approach A (2-Phase Job) 채택

## 1. 현재 문제

| 문제 | 원인 |
|------|------|
| 프리셋이 너무 많아서 뭘 골라야 할지 모름 | 11개 프리셋을 사용자가 직접 선택 |
| 프리셋과 템플릿의 관계가 불명확 | 두 선택이 각각 뭘 결정하는지 UI에서 안 보임 |
| 보고서 품질이 요약보다 낮음 | NotebookLM 보고서 생성은 briefing_doc 형식 고정 |
| 템플릿이 실제로 보고서에 영향 없음 | templateConfig가 슬라이드에만 반영 |

## 2. 새 플로우 설계

### 3단계 플로우: 업로드 → 프리셋 선택 → 템플릿 선택

```
[1. 파일 업로드]
    ↓ 백그라운드: NotebookLM 업로드 + 분석 시작 (비동기)
    ↓ 즉시 Step 2로 전환 (대기 없음)

[2. 프리셋 선택] ← 전체 프리셋 즉시 표시
    ↓ AI 분석 완료 시 → 추천 프리셋에 ⭐ 뱃지 실시간 추가
    ↓ 사용자는 추천을 기다리지 않고 바로 선택 가능
    ↓ 사용자 선택 → 산출물 성격/목적 결정

[3. 템플릿 선택] ← 슬라이드 스타일만 결정
    ↓ 색상/폰트/톤앤매너/레이아웃
    ↓ 슬라이드가 없는 프리셋이면 이 단계 건너뜀

[완료] → 요약 + 슬라이드 생성 시작
```

> **핵심 UX 원칙**: 사용자를 절대 대기시키지 않는다.
> AI 분석은 백그라운드에서 진행되고, 결과가 오면 UI가 실시간 업데이트된다.
> 분석이 안 끝났어도 사용자는 직접 프리셋을 골라서 진행할 수 있다.

### 핵심 원칙
1. **산출물 = 요약(텍스트) + 슬라이드(PDF)** — 보고서(NotebookLM briefing_doc)는 제거
2. **프리셋 = 요약의 구조와 목적 결정** — "이 자료를 어떤 관점으로 정리할 것인가"
3. **템플릿 = 슬라이드의 비주얼 스타일** — "어떤 톤으로 보여줄 것인가"
4. **AI 추천 = 사용자 부담 제거** — 자료를 읽고 맞는 프리셋만 추천

## 3. Step 1: 파일 업로드 + 자동 분석

### UI
- 현재와 동일한 드래그&드롭 업로드
- 업로드 완료 즉시 → **바로 Step 2 (프리셋 선택) 화면으로 전환**
- 백그라운드에서 분석 진행, 완료 시 프리셋 목록에 추천 뱃지 실시간 추가
- 분석 완료 시 상단에 분석 결과 카드 표시 (선택 사항, 사용자가 이미 선택했으면 무시)

### 백엔드 처리
```
1. 파일 → Worker 업로드 (즉시 응답: jobId + filePath)
2. 백그라운드: NotebookLM 노트북 생성 + 소스 추가 (30-120초)
3. 백그라운드: 분석 질문 → 자료 유형 파악
4. Job.analysisResult에 결과 저장, status → "analyzed"
5. 프론트가 polling으로 분석 결과 수신 → UI 업데이트
```

## 6. 산출물 변경

### Before (현재)
| 산출물 | 소스 | 문제 |
|--------|------|------|
| 요약 (텍스트) | NotebookLM chat.ask | 품질 좋음 |
| 보고서 (텍스트) | NotebookLM generate_report | 품질 낮음, 형식 고정 |
| 슬라이드 (PDF) | NotebookLM generate_slide_deck | 타임아웃 이슈 있었음 |

### After (v2)
| 산출물 | 소스 | 설명 |
|--------|------|------|
| **요약** (텍스트) | NotebookLM chat.ask + 프리셋 프롬프트 | 자료 목적에 맞는 구조화된 요약 |
| **슬라이드** (PDF) | NotebookLM generate_slide_deck + 템플릿 | 비주얼 스타일 반영된 슬라이드 |

- **보고서 제거** — 요약이 보고서 역할을 대신함
- 프리셋이 요약의 구조와 관점을 결정하므로, "임원 보고" 프리셋으로 생성한 요약 = 임원 보고서

## 7. 백엔드 변경 요약

### 새로운 Job 처리 플로우

```
[Phase 1: 분석] — Job 생성 시 즉시 실행
  1. 노트북 생성 + 소스 추가
  2. 기본 분석 질문 → 자료 유형 파악
  3. Job에 분석 결과 저장 (analysisResult)
  4. status: "analyzed" (새 상태)
  → 프론트에서 분석 결과 표시 + 프리셋 추천

[Phase 2: 생성] — 사용자가 프리셋+템플릿 선택 후
  1. 요약 생성 (chat.ask + 프리셋 프롬프트)
  2. 슬라이드 생성 (generate_slide_deck + 템플릿 스타일)
  3. status: "complete"
```

### DB 변경
```python
class Job:
    # 기존 필드 유지
    # 추가 필드
    analysisResult = Column(Text, nullable=True)   # JSON: {type, topic, summary, suggestedPresets}
    # reportPath 제거 (또는 deprecated)
```

### API 변경
```
POST /api/jobs/analyze     — 파일 업로드 + 분석 (Phase 1)
POST /api/jobs/{id}/generate — 프리셋/템플릿 선택 후 생성 시작 (Phase 2)
GET  /api/jobs/{id}         — 기존 유지
GET  /api/jobs/{id}/download/{type} — summary, slides만 (report 제거)
```

## 8. UI/UX 개선 방향

### 디자인 원칙
1. **한 화면에 하나의 선택** — 스크롤 최소화
2. **AI가 먼저 제안** — 사용자는 확인/수정만
3. **진행 상태 시각화** — 각 단계 완료 시 체크 표시
4. **모바일 퍼스트** — 텔레그램에서 링크 열어도 잘 보이게

### 시각적 변경
- Step 1 (업로드): 풀스크린 드래그 영역, 분석 중 애니메이션
- Step 2 (프리셋): 카드형, 추천 뱃지, 접기/펼치기
- Step 3 (템플릿): 색상 프리뷰 그리드, 선택 시 미리보기
- Step 4 (확인): 선택 요약 + "생성 시작" 버튼만
- Job 상세: 요약 인라인 표시 + 슬라이드 다운로드 버튼

- **논블로킹 분석**: 사용자는 대기 없이 즉시 프리셋 선택 가능, AI 추천은 비동기로 도착 시 UI 업데이트

- **빠른 시작**: 분석 건너뛰고 프리셋 전체 목록에서 선택하는 옵션

## 10. 분석 프롬프트 설계

```
이 자료의 종류와 핵심 주제를 JSON 형식으로 알려주세요.
종류는 다음 중 하나를 선택하세요:
- meeting_regular (정기회의, 주간/월간 미팅)
- meeting_strategy (전략/의사결정 회의)
- meeting_external (고객/파트너 외부 미팅)
- meeting_tech (기술/개발 회의)
- seminar (강연, 세미나, 교육)
- brainstorming (아이디어 발산, 기획)
- project (프로젝트 관리, 진행 점검)
- directive (지시, 전달, 공지)
- report (보고, 브리핑)
- general (기타)

응답 형식:
{"type": "meeting_strategy", "topic": "2025년 하반기 BaaS 사업 전략", "summary": "한 줄 요약"}
```
